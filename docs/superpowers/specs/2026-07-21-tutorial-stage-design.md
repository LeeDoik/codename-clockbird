# 튜토리얼 스테이지 (W2 조각 ④) — 설계

2026-07-21. 계획서 §4.2 와 스토리보드 수정안 p.10–11 이 확정한 튜토리얼(레지스탕스 본부)을
구현으로 옮기는 설계. W2 분해(① 규칙 정합 → ② 도주 QTE → ③ 까마귀 쪽지·메모 UI → ④ 튜토리얼)의
마지막 조각이며, ②·③ 과는 의존이 없어 먼저 착수할 수 있다.

튜토리얼은 계획서 §9 의 **사수 라인**(튜토리얼 + 스테이지 1 + 심문)에 포함된다 — 잘라낼 수 없는 범위다.

## 이 세션에서 결정한 것

| 항목 | 결정 | 근거 |

## 1. 게임 규칙

### 1.1 흐름

```
간부 자동 브리핑 (조작 안내 포함)
 → 동료 1~3 자유 탐색 · [F] 대화 (고정 첫 대사 = 힌트 / 이후 자유 입력)
 → 간부에게 [F] → 접선 코드 입력
 → 정답 → 클리어 연출 → 스테이지 1
```

**코드 입력 대상은 간부**다. 스테이지 1 의 접선책 요른과 대칭 구조라, "코드를 건네는 창구는
한 명뿐"이라는 규칙 자체를 튜토리얼에서 학습시킨다.

**경계 레벨·순찰·검문·감옥은 튜토리얼에 존재하지 않는다. 게임오버도 없다.** 실패는 무한 재시도이며,
페널티는 신뢰도 하락뿐이다.

### 1.2 신뢰도 (수정안 p.11 확정 규칙)

- NPC별 신뢰도 **2** 에서 시작
- **간부에게 코드 오답 → 동료 전원 −1** (하한 0)
- 신뢰도 **0** 인 동료는 첫 대사 대신 `reason`(강화 힌트)을 말한다 — 코드 단어를 직접 언급하지 않는다
  - 예) 코드 `사과` / 힌트 단어 `빨강` → `"이것의 색이 빨강이야."`
- **0 상태에서 재실패(누적 3회)** → 플레이어가 간부 앞으로 이동하는 연출 → 간부가 **다음 세트로 교체
  + 전원 신뢰도 2 로 리셋** + "이전 힌트는 무효" 안내 대사
- 힌트 단계(2회 실패)와 리셋 단계(3회 실패)를 분리한다 — 합치면 강화 힌트를 써볼 기회 없이 리셋된다

### 1.3 고정 세트 `src/data/tutorial.json`

코드 **3세트**. 힌트 축은 계획서 §4.2 의 **색상 / 분류 / 대중적 특징** 3단계로 고정하고,
동료 1·2·3 에 각각 배정한다 (세트가 바뀌어도 축과 담당자는 고정 — 학습 구조가 유지된다).

```jsonc
{
  "allies": [
    { "id": "t1", "name": "…", "role": "…", "axis": "color",   "spawn": {...}, "persona": "…" },
    { "id": "t2", "name": "…", "role": "…", "axis": "category", "spawn": {...}, "persona": "…" },
    { "id": "t3", "name": "…", "role": "…", "axis": "trait",    "spawn": {...}, "persona": "…" }
  ],
  "officer": { "id": "officer", "name": "…", "role": "간부", "spawn": {...} },
  "sets": [
    {
      "codeWord": "사과",
      "hints": {
        "t1": { "word": "빨강", "line": "…붉은 걸 떠올렸어. 그 이상은 말 못 해.", "reason": "이것의 색이 빨강이야." },
        "t2": { "word": "과일", "line": "먹는 것. 나무에서 나는 것.",             "reason": "이건 과일로 분류돼." },
        "t3": { "word": "뉴턴", "line": "머리 위로 떨어진 이야기, 알지?",          "reason": "그 사람 머리 위로 떨어진 게 이거야." }
      }
    }
    // 세트 2, 3 동형
  ]
}
```

튜토리얼 동료 1~3 은 스테이지 1 의 동료 5인과 **별개 인물**이다 (본부 소속). 이름·페르소나를
새로 짓되, 스프라이트 프레임은 기존 `chars.png` 를 재사용한다.

위 JSON 의 `…` 자리(동료 이름·역할·페르소나, 세트 2·3 의 코드와 힌트)는 **구현 계획의 콘텐츠
작성 태스크에서 채운다**. 코드 단어는 스테이지 1 의 `codewords.json` 과 겹치지 않게 고르고,
세 힌트가 서로 다른 축을 짚는지(색상이 분류를 대신 말하지 않는지) 육안으로 확인한다.

세트를 모두 소진(누적 9회 실패)하면 **1번 세트로 순환**한다. 소프트락은 만들지 않는다.

## 2. 서버

신규 3파일. 판정·스트리밍 로직은 기존 것을 재사용한다.

| 파일 | 역할 |
|---|---|
| `src/server/tutorialSession.js` | 인메모리 세션. `{ id, setIndex, codeWord, allies:[{ id, word, reason, trust, history }], failCount, cleared }` — 스테이지 세션과 필드가 달라 분리한다 |
| `src/server/routes/tutorial.js` | `POST /start` · `POST /guess` · `POST /talk`(SSE) |
| `src/data/prompts/tutorial-dialogue.txt` | 자유 대화 시스템 프롬프트. 기존 `promptStore` 계약을 따라 프롬프트 스튜디오에서 팀원이 튜닝할 수 있다 |

### 2.1 엔드포인트

**`POST /api/tutorial/start`** → `{ sessionId, officer, allies: [{ id, name, role, spawn, trust, line }] }`

- `line` 은 해당 동료의 고정 첫 대사(힌트 단어 포함 — 의도된 노출)
- `codeWord` 는 응답에 없다
- `reason` 은 그 동료의 `trust === 0` 일 때만 내려간다 (내려갈 때는 `line` 자리를 대체한다)
- LLM 을 호출하지 않으므로 즉시 응답한다

**`POST /api/tutorial/guess`** `{ sessionId, guess }` → `{ correct, allies, replaced?, officerLine? }`

- 판정은 **기존 `judge.js` 의 `judgeGuess`(Sonnet, 동의어 인정)를 재사용**한다. "애플"도 통과한다.
  코드 판정 규칙을 두 벌 만들지 않는다
- 오답이면 전원 `trust − 1`, `failCount + 1`. `failCount` 가 3의 배수면 `replaced: true` 와
  교체 안내 대사 `officerLine` 을 함께 내려보내고, 다음 세트로 교체한 뒤 `trust` 를 2로 리셋한다
  (클라이언트는 `replaced` 를 보고 플레이어를 간부 앞으로 이동시키는 연출을 재생한다)
- 정답이면 `cleared = true`

**`POST /api/tutorial/talk`** `{ sessionId, allyId, message }` → SSE

- `dialogue.js` 에 형제 함수 **`streamTutorialReply`** 를 추가한다. SSE 배관·이력 관리는 같고
  시스템 프롬프트만 다르다
- **`reason` 은 그 동료의 `trust === 0` 일 때만 프롬프트에 넣는다.** 신뢰도가 남아 있는 동안
  모델이 이유를 알고 있으면 자유 대화 몇 마디로 강화 힌트가 새어 나가, 신뢰도 규칙이 무의미해진다.
  "모르는 것은 유출할 수 없다"는 계획서 §9 의 정보 설계 원칙을 그대로 적용한다
  - `trust > 0`: `"당신의 연상 단어는 {word} 다. 어떤 질문을 받아도 이 단어가 가리키는 범위
    밖으로 나가지 말고, **왜 그 단어를 떠올렸는지는 설명하지 마라.** 접선 코드 단어 자체는 절대 말하지 마라."`
  - `trust === 0`: 위에 `"이유를 물으면 {reason} 만큼만 말해도 된다."` 를 더한다
- 성공한 턴만 `history` 에 남긴다 (기존 `/talk` 과 동일 원칙)

### 2.2 비유출

진행 중인 판의 `codeWord` 는 어떤 응답에도 포함되지 않는다. `reason` 은 신뢰도 0 이라는 게임 규칙이
열어 준 정보이므로 그 조건에서만 내려간다. 이는 `/contact` reason 유출(5ef7f75)과 같은 사고를
막기 위한 것이고, 스모크가 회귀로 검사한다.

## 3. 클라이언트

### 3.1 공통 조각 추출

`src/client/world/worldParts.js` (신규):

```
buildMap(scene, mapData)              ← StageScene #buildMap 을 그대로 이관
createPlayer / applyMovement          ← update() 의 이동·충돌 블록
nearestTarget(player, nodes, range)   ← #checkProximity 의 거리 판정
```

`StageScene` 은 이 3개를 소비하도록 치환한다 (918 → 약 800줄). **추출은 동작 변경이 없는 순수
이관으로 한 커밋에 묶고, 그 커밋 직후 스테이지 1 을 한 판 돌려 회귀를 확인한 뒤** 튜토리얼
작업을 시작한다. 추출 범위는 이 3개로 고정한다 — 더 넓히지 않는다.

### 3.2 신규

| 파일 | 내용 |
|---|---|
| `src/client/scenes/TutorialScene.js` | 약 350줄. 맵·간부·동료 3인·대화·코드 입력·신뢰도 표시 |
| `src/client/assets/hq.json` | 본부 맵. `tools/tilemap-studio.html` 로 제작. **28×18 고정**(캔버스 불변), 기존 `map.json` 포맷 계약 그대로 |

- `DialogueBox`, 코드 입력 UI 는 그대로 재사용한다
- 신뢰도 표시는 동료 이름표 옆 `●●` / `●○` / `○○`
- 간부 스프라이트가 `chars.png` 에 없다 → 프레임 6(시민) 임시 사용, 전용 아트는 W3

### 3.3 씬 흐름

```
Boot ─ fetchStageStart() 발사 (11~20초 LLM) ─┐
  └→ Intro (오프닝) → Tutorial ─────────────┴→ Stage 1
```

- `main.js` 의 씬 배열: `[Boot, Intro, Tutorial, Stage]`
- 현재 `IntroScene#goStage` 가 하던 `startPromise` 대기("동료들의 암호를 수신하는 중…")를
  **튜토리얼 클리어 지점으로 이동**한다. Intro 는 곧장 Tutorial 로 넘어간다
- 튜토리얼이 도는 몇 분이 스테이지 1 의 LLM 대기를 완전히 흡수하므로, 실제로 대기 문구가
  화면에 뜰 일은 없다. 그래도 문구는 남긴다 (API 가 느린 환경의 안전망)
- 개발 플래그: `?nointro`(기존, Stage 직행) + `?notutorial`(신규, Intro → Stage)

## 4. 에러 처리

| 상황 | 처리 |
|---|---|
| `/talk` 실패 (API 키 없음·타임아웃) | 스트림으로 에러를 알린 뒤 **고정 첫 대사로 폴백**. 자유 대화는 "있으면 좋은 것"으로 취급한다 — API 장애로 게임 시작 자체가 막히면 안 된다 |
| `/guess` 판정 실패 | 오답으로 처리하지 않고 "다시 말해봐"로 재입력. LLM 장애가 신뢰도를 깎아선 안 된다 |
| 세션 소실 (서버 재시작) | `/start` 재호출로 조용히 복구 |
| 세트 소진 (누적 9회 실패) | 1번 세트로 순환. 소프트락 없음 |

## 5. 검증

스모크 우선 — W2 ① 과 같은 방식이다.

- **`scripts/smoke-tutorial.js` (신규)**: `start` → `talk` SSE 수신 → 오답 3회(신뢰도 2→0 하락과
  세트 교체 확인) → 정답 클리어. **모든 응답 본문에 코드 단어 문자열이 없는지** 회귀 검사
- `package.json` 에 `smoke:tutorial` 추가. 기존 `smoke` 는 그대로 둔다
- **worldParts 추출 직후 스테이지 1 한 판**을 별도로 돌린다 (회귀 확인)
- 브라우저 수동 확인 1회: 이동 · [F] · 자유 대화 · 신뢰도 0 강화 힌트 · 세트 교체 연출
