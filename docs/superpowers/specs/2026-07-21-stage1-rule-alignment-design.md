# 스테이지 1 규칙 정합 (W2 조각 ①) — 설계

2026-07-21. 계획서 §4.6 조정 목록의 #2(접선책 단일 입력)·#3(경계 3단계)·#4(신뢰도 존폐)를
구현에 반영하는 설계. W2 분해(① 규칙 정합 → ② 도주 QTE → ③ 까마귀 쪽지·메모 UI → ④ 튜토리얼)의
첫 조각이다.

## 확정된 결정 (이 세션에서 결정, 문서 반영 필요)

| 항목 | 결정 |
|---|---|
| 스테이지 1 신뢰도·밀고 | **제거**. 코드 입력 실패 = 경계 +1. 신뢰도 개념은 튜토리얼(신뢰도 2)에만 남긴다 |
| 구출 시 경계 +1 | **유지** ("구출 소동"). 스토리보드 p.16에 "성공 시에도 소동으로 경계 +1" 명시 추가 |
| 경계 레벨 | **초기 0, 상한 3**. 레벨 2 증원 / 레벨 3 강화 + 발각 즉사 |
| 작품명 | 보류 — 팀 결정 후 반영 (이 스펙 범위 아님) |

신뢰도 제거의 근거: 접선책 단일 입력(#2)과 현행 밀고 규칙이 결합하면 "실패 3회 → 접선책 밀고 →
코드 입력 창구 소멸 = 정답을 알아도 패배"라는 즉사 경로가 생긴다. 실패 페널티를 경계 +1로
전환하면 리스크 축이 경계 레벨 하나로 정리되고, 레벨 3 즉사 규칙과 자연스럽게 맞물린다.

## A. 신뢰도·밀고 제거 (서버)

- `src/server/session.js`
  - 동료 상태에서 `trust`·`informed` 필드 제거. `MAX_TRUST`·`RESCUED_TRUST` 상수 제거.
  - `loseTrust()`·`informedCount()`·`isUnwinnable()` 삭제.
    - `isUnwinnable` 삭제 근거: 밀고가 사라지면 영구 접선 불가 상태가 없다. 전원 체포여도
      구출 활로가 항상 열려 있어 unwinnable 상태 자체가 소멸한다. 호출부 분기도 함께 제거.
  - `rescueAlly()`: 신뢰도 하락(`RESCUED_TRUST`) 로직만 제거. **경계 +1은 유지.**
  - `toClientView()`: `trust`/`maxTrust`/`informed` 내보내기 제거. 단어 노출 조건에서
    `informed` 검사 제거 (`contacted && !arrested` 만 남는다).
- `src/server/routes/stage.js`
  - 코드 입력 실패 처리: `loseTrust` 호출 → `raiseAlert(session)` 로 교체.
    응답 페이로드 `{ informed, trust }` → `{ alertLevel }`.
- 클라이언트: 신뢰도 칸·밀고 상태를 그리던 UI 제거 (수첩·HUD·대사 등, 접점은 조사 결과 기준).

## B. 경계 레벨 3단계 (서버 + 클라)

- 초기 0 (현행 유지). 상한 4→3: `src/client/entities/Patrol.js`·`src/client/minigames/timingLock.js`
  의 `MAX_LEVEL` 조정. 상수가 두 파일에 중복돼 있으므로 공용 상수로 뽑을지는 구현 계획에서 판단.
- 레벨별 효과:
  - 레벨 1: 현행 (순찰 가속 축 유지)
  - 레벨 2: 증원 — 구체 수단(순찰 로봇 수·속도)은 현재 레벨 연동 방식 조사 결과에 맞춰 계획에서 확정
  - 레벨 3: 강화 + **발각 즉사** — 발각 시 검문(현행)에 들어가지 않고 즉시 게임오버.
    이 분기는 발각 처리의 앞단 규칙이므로, W2 ②에서 검문이 도주 QTE로 교체돼도 그대로 유지된다.

## C. 접선책 신설 + 코드 입력 단일화

- 접선책(시계 수리공) NPC를 동료 5인과 **별도로** 추가 (수정안 §4 #7: "동료 5 + 접선책 1").
  - 연상 단어를 제출하지 않는다 → 중복 판정·체포 대상이 아니다.
  - 세션 상태에서 동료 배열과 분리해 표현한다 (동료용 필드 `word`/`arrested` 등을 억지로 공유하지 않는다).
- 코드 입력 진입점 단일화:
  - 클라: 동료 접선(`#offerCode`) 후 열리던 코드 입력창 제거. 접선책과 상호작용할 때만 입력 UI.
    `/contact` 는 "단어 공개" 전용이 된다.
  - 서버: 코드 검증 라우트가 접선책 대상임을 검증, 아니면 4xx. 클라만 막으면 API 직접 호출로
    우회 가능하므로 서버에서도 막는다 — `/alarm` 화이트리스트(`ALARM_REASONS`)와 같은 원칙.

## D. 검증

- `scripts/smoke-talk-sse.js` 확장: 코드 실패 → `alertLevel` 상승 확인, 응답에
  `trust`/`informed` 부재 확인 (기존 codeWord·reason 비유출 체크와 같은 패턴).
- 회귀 확인: 접선 → 단어 공개, 구출 → 경계 +1, 접선책 아닌 대상 코드 입력 → 4xx.
